### **SYSTEM PROMPT / DATABASE CONTEXT**

Here is the conceptual Entity-Relationship Diagram (ERD) for my database. Please use this schema context for any SQL queries, database design, or logic I ask for.

**Entities & Attributes:**

1. **Restaurants**
* `restaurantID` (int) - **Primary Key**
* `name` (nvarchar(100))
* `address` (nvarchar(100)) - **Composite attribute** consisting of:
* `street` (nvarchar(50))
* `city` (nvarchar(50))




2. **Employees**
* `empID` (int) - **Primary Key**
* `FullName` (nvarchar(50))
* `gender` (char(1))
* `phone` (nvarchar(20)) - **Multi-valued attribute** (represented by a double oval, meaning one employee can have multiple phone numbers. This will require a separate table in the physical schema).


3. **Shifts**
* `shiftID` (int) - **Primary Key**
* `shiftDate` (date)
* `startTime` (time)
* `endTime` (time)



**Relationships:**

* **has (1:N):** One `Restaurant` (1) has many `Employees` (N). There is a double line from Employees to the 'has' relationship, indicating total participation (every employee must belong to a restaurant).
* **works (N:M):** Many `Employees` (N) work many `Shifts` (N). This is a Many-to-Many relationship (which implies a bridging table is needed in the physical database implementation).
